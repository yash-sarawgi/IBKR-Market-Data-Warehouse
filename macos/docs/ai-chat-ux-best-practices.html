AI Chat UX Best Practices for Market Data Warehouse

This report finishes the missing assistant UX research for the native macOS client. It combines current Apple macOS guidance with provider-side prompting guidance from OpenAI, Anthropic, and Google, then maps the findings onto the live Market Data Warehouse shell.

Product Reality

Chat is only one lane

This app is not a generic chatbot. It is a local-first parquet and DuckDB workstation where AI assistance must coexist with raw SQL and raw duckdb commands.

Primary UX Rule

Preserve trust surfaces

Users need to know which provider answered, which auth path is active, what command will run, and what actually executed.

Native macOS Rule

Use the platform

Settings, menus, split views, tables, keyboard shortcuts, file panels, and accessibility should feel Mac-native instead of web-like.

Operational Rule

Design for failure

CLI not installed, login missing, API key missing, no source selected, and provider rate limits are normal states and need explicit recovery paths.

Summary

The strongest cross-source pattern is consistent: assistant UX works best when the product is transparent, scoped, and user-controlled. Apple emphasizes clear disclosure and native affordances. OpenAI, Anthropic, and Google all emphasize prompt clarity, structured context, and evaluation against real tasks. For this app, that translates into a chat surface that never hides the real DuckDB path and never makes provider/auth state ambiguous.

The assistant should accelerate local data work, not replace the user's ability to inspect, edit, rerun, and debug the exact SQL or DuckDB command path.

User Story

As a macOS researcher exploring local parquet files and DuckDB databases, I want a native assistant that can explain, propose, and run data operations while keeping the exact execution path visible, so I can move faster without giving up control or debuggability.

High-Signal Principles

Principle Why it matters here Practical implication
Separate AI output from system action Trust drops if the app blurs advice with execution. Keep assistant text, SQL previews, raw duckdb previews, and command results visually distinct.
Make provider/auth state visible This app supports local subscription CLIs plus API-key fallback. Show provider name, auth mode, and recovery state in setup, settings, and diagnostics.
Prefer progressive setup First-run setup should remove dead ends, not create a wizard maze. Default provider, auth mode, optional model, and secure key entry are enough for v1.
Keep deterministic escape hatches Data tools need precise control when chat abstraction fails. Preserve /sql ..., /duckdb ..., copyable output, and rerunnable commands.
Version prompts and evaluate changes Small wrapper-prompt changes can regress the assistant. Keep structured prompt wrappers, representative fixtures, and regression checks tied to provider UX work.

Native macOS Guidance Applied

Setup and settings

  • Initial setup should block the main shell only once, then reappear as a sheet or Settings scene for reconfiguration.
  • The standard macOS Settings scene should remain authoritative. If the app mirrors those controls inside the workspace, both views must bind to the same state store.
  • Secrets belong in Keychain. Ordinary settings and session state belong in Application Support.

Conversation and navigation

  • Use NavigationSplitView for the shell, a bottom composer, and native toolbar and menu commands for frequent actions.
  • Use suggestions for high-probability actions, but keep them secondary to manual control.
  • Plan for Table-based result presentation, search, and an optional inspector for schema, query plans, and command metadata.

Files and persistence

  • Continue using native file panels and drag-and-drop for source import.
  • If the app moves toward sandboxing or wider file retention, replace raw file URLs with security-scoped bookmarks.
  • Preserve recent sources, selected source, provider defaults, and transcript state across launches, but keep secret material out of the session snapshot.

Provider and Prompting Guidance Applied

  • OpenAI, Anthropic, and Google all recommend specific instructions, strong context framing, and examples where ambiguity matters. For this app, the wrapper prompt should always include the current source, a short transcript tail, and the user's intent.
  • Assistant responses should stay concise, but when a data operation is implied, the response should prefer an explicit SQL or DuckDB command over vague narrative advice.
  • Keep prompt wrappers small and inspectable. This product benefits more from precise scaffolding than from long hidden system prose.
  • Prompt changes should be evaluated against a stable set of data-work tasks: no source selected, parquet preview, schema inspection, row count, raw SQL, raw DuckDB flags, missing CLI, and missing API key.

Implications for the Current App

Current area Keep Next research-backed refinement
First-run setup Block on first launch, rerun later from settings. Keep the initial setup path short and add clearer provider/auth recovery copy for missing CLI login vs missing API key.
Settings Dedicated Settings scene plus in-app mirror is acceptable when both use the same store. Use the Settings scene as the canonical macOS preference surface and keep the workspace mirror task-oriented.
Assistant transcript Separate assistant, user, preview, and result states. Add faster retry, copy, and rerun affordances before expanding message complexity.
DuckDB parity Raw command visibility already aligns with trust guidance. Keep parity language explicit in docs and future UI labels so users never mistake helper SQL for the whole surface.
Provider execution CLI-backed provider calls are practical, but vendor auth and flag contracts can drift. Re-verify exact CLI flags and native OAuth options during implementation changes.

Acceptance Criteria

  • The app always shows which provider answered and whether the current path depends on local CLI auth or an API key.
  • Every executed SQL or raw DuckDB invocation remains visible, copyable, and distinguishable from assistant prose.
  • First-run setup is short, recoverable, and can be rerun without leaving the app shell.
  • Settings remain consistent between the macOS Settings scene and any in-content settings surface.
  • Provider and command failures produce actionable recovery text instead of generic chat errors.
  • Prompt-wrapper changes are evaluated against representative data tasks before shipping.

Non-goals

  • Turning the app into a generic multi-agent chat client.
  • Hiding DuckDB behind purely conversational abstractions.
  • Building a custom web-style settings system when native macOS scenes and forms are available.
  • Adding broad write workflows before the read-first exploration path is fully hardened.

Risks and Assumptions

  • Provider CLI behavior can change faster than the app's release cycle.
  • Native OAuth remains a future path; current UX research assumes the local CLI path is the primary production flow.
  • Future sandboxing would require bookmark-based file persistence instead of plain file URLs.
